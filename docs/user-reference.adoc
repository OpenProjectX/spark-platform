= Spark Platform User Reference
:toc: left
:toclevels: 3
:sectanchors:

Spark Platform provides a Gradle plugin, a BOM, and a Jib-built platform image
for Spark applications.

The main goal is to make Spark runtime ownership explicit. Application projects
select a Spark line, runtime variants, and optional addons; Spark Platform owns
the Spark, Hadoop, Scala, variant, and addon runtime artifacts for that
selection. This keeps local, CI, and production builds on the same stack and
avoids application builds that work locally only because a developer happened
to declare a different Spark dependency.

== Concepts

Spark line::
  A named Spark compatibility line, such as `spark3` or `spark4`.
  The line selects version-catalog bundles named
  `spark-platform-<line>-managed`.

Platform-owned dependency::
  A Spark, Hadoop, Scala, variant, or addon runtime artifact whose version is
  owned by Spark Platform. Application builds may opt into these coordinates on the
  `sparkPlatform` configuration, but they do not declare versions for them.

Application-owned dependency::
  A library that belongs to the application itself and is not part of the Spark
  runtime platform. Application builds continue to declare these dependencies
  on normal Gradle configurations such as `implementation`.

Official build::
  A build where Spark Platform dependencies are treated as `compileOnly`.
  This is intended for images that run on top of the platform base image.

Local build::
  A normal developer build where Spark Platform dependencies are treated as
  `implementation`, so the application can run from Gradle or an IDE.

Platform image::
  A container image built by the `platform-image` module. It places selected
  variant and addon jars under `/opt/spark/jars` on top of the matching Apache
  Spark base image.

Spark base image::
  A project-owned image built by `spark-base-image`, published as
  `ghcr.io/openprojectx/spark:<spark-version>-scala<scala>-...`. It owns the
  Spark line runtime baseline: Spark, Scala, Hadoop, and line-managed runtime
  jars such as Spark SQL Kafka and Spark Avro.

Application image::
  A user application image built by Jib from the selected platform image. It
  contains application classes and application-owned libraries. Platform-owned
  Spark jars are provided by the parent image instead of being repackaged into
  the application image.

== Applying the Plugin

[source,kotlin]
----
plugins {
    java
    id("org.openprojectx.spark.platform")
}
----

The plugin creates:

* the `sparkPlatform` extension
* the `sparkPlatform` dependency configuration
* the `sparkPlatformBom` dependency configuration, reserved for explicit BOM use
* the `sparkPlatformJavaExecRuntime` dependency configuration for CI smoke runs
* Jib configuration for application images

The plugin expects a Gradle version catalog named `libs`. The repository uses
`gradle/libs.versions.toml` as the source of truth for Spark Platform versions.

== Dependency Ownership Contract

Application projects select the platform:

[source,kotlin]
----
sparkPlatform {
    line.set("spark4")
    variants.set(listOf("iceberg"))
    addons.set(listOf("hadoopAws"))
    managedConfigurations.set(listOf("api", "testImplementation"))
}
----

The plugin then adds the platform-owned dependencies from
`gradle/libs.versions.toml` as strict constraints. For the example above, the
selected constraint bundles include Spark 4, Hadoop for Spark 4, and the Spark 4
Iceberg runtime. Addons such as `hadoopAws` add platform-owned supplemental
libraries without becoming Spark/Scala compatibility dimensions.

Managed bundles are a Gradle constraint contract. They do not copy jars into an
application image by themselves. A managed coordinate affects dependency
resolution only when the application also declares that coordinate, either with
`sparkPlatform(...)` or through a normal Gradle configuration listed in
`managedConfigurations`.

Application builds then opt into the platform-owned APIs they compile against by
using the `sparkPlatform` configuration without versions:

[source,kotlin]
----
dependencies {
    sparkPlatform("org.apache.spark:spark-sql_2.13")
    sparkPlatform("org.apache.iceberg:iceberg-spark-runtime-4.0_2.13")
}
----

When the platform should behave like a BOM for a normal Gradle scope, point
`managedConfigurations` at that scope and use normal versionless dependencies
there:

[source,kotlin]
----
plugins {
    `java-library`
    id("org.openprojectx.spark.platform")
}

sparkPlatform {
    line.set("spark4")
    variants.set(listOf("iceberg"))
    managedConfigurations.set(listOf("api", "testImplementation"))
}

dependencies {
    api("org.apache.spark:spark-sql_2.13")
    testImplementation("org.apache.iceberg:iceberg-spark-runtime-4.0_2.13")
}
----

Application builds should not put versions for Spark, Hadoop, Scala, Iceberg,
Hudi, Paimon, or OpenLineage artifacts in build scripts. To change that stack,
change the platform catalog and platform image, then let applications select the
updated line, variant, or addon. Variants describe compatibility-bearing Spark
runtime families. Addons describe supplemental platform-owned dependency packs.
Curated platform images use profile tags such as `spark4-lakehouse-<version>`
so common addons do not make tags grow without bound.

Application-owned libraries still use normal Gradle configurations:

[source,kotlin]
----
dependencies {
    implementation("com.example:business-client:1.2.3")
}
----

=== Managed Versions Versus Packaging Scope

Managed versions and packaging scope are separate concerns.

Any configuration listed in `managedConfigurations` extends `sparkPlatform`, so
dependencies declared there can omit versions when the coordinate is managed by
the selected line, variant, or addon. For example, this is valid:

[source,kotlin]
----
sparkPlatform {
    line.set("spark4")
    variants.set(listOf("iceberg"))
    managedConfigurations.set(listOf("api", "testImplementation"))
}

dependencies {
    api("org.apache.spark:spark-sql_2.13")
    testImplementation("org.apache.iceberg:iceberg-spark-runtime-4.0_2.13")
}
----

`sparkPlatform(...)` is the plugin's dedicated provided-platform scope. It has
two jobs:

* it declares that the application compiles against that platform-owned API
* it lets the platform constraint choose the version

Normal Gradle scopes keep their normal packaging behavior. If `implementation`
is listed in `managedConfigurations`, then an `implementation(...)` dependency
can omit its version and will still be packaged into the application runtime
classpath by Gradle/Jib.

That is useful for application-owned libraries whose versions are intentionally
managed by the platform. It should not be the default for Spark-owned runtime
jars. For Spark, Hadoop, variants, and addons, use `sparkPlatform(...)` unless
you are deliberately packaging that jar into the application image and accepting
the duplicate-classpath risk.

Class-not-found behavior follows the same rule:

* If the app declares `sparkPlatform("org.apache.spark:spark-sql-kafka-0-10_2.13")`
  on `spark4`, the app image expects the Spark base image to provide that jar.
* If the app declares an Iceberg API with `sparkPlatform(...)`, the app must
  select `variants.set(listOf("iceberg"))` and run on a platform image built
  with Iceberg.
* If the selected image does not contain the matching platform-owned jar, the
  app will fail at runtime. That is an image/platform mismatch, not something
  the user should fix by adding a versioned `implementation` dependency.

== Local Versus Official Builds

For local builds, `sparkPlatform` and `sparkPlatformBom` extend
`implementation`. The plugin adds strict constraints to `sparkPlatform`, and
the user-selected `sparkPlatform(...)` dependencies become local runtime
dependencies:

[source,text]
----
implementation
  extendsFrom sparkPlatform, sparkPlatformBom
----

For official builds, they extend `compileOnly`:

[source,text]
----
compileOnly
  extendsFrom sparkPlatform, sparkPlatformBom
----

Managed `JavaExec` tasks also receive a resolvable
`sparkPlatformJavaExecRuntime` classpath in official mode. This keeps JVM smoke
tests runnable in CI without changing application image packaging: the
application image still treats platform-owned dependencies as provided by the
platform base image.

Packaging behavior:

* Local JVM runs: selected `sparkPlatform(...)` dependencies are available on
  the local runtime classpath so the app can run from Gradle or an IDE.
* Official application images: selected `sparkPlatform(...)` dependencies are
  treated as provided and are not packaged into the application image.
* Platform-owned runtime jars are packaged in the Spark base image or platform
  image layers, depending on whether they are line baseline content or selected
  variants/addons.

Official mode is enabled when any of these are true:

* `-PsparkPlatform.officialBuild=true`
* `CI` environment variable is present
* `GITHUB_ACTIONS` environment variable is present
* `JENKINS_HOME` environment variable is present

== Plugin Extension

Configure the plugin with the `sparkPlatform` extension.

[source,kotlin]
----
sparkPlatform {
    line.set("spark4")
    profile.set("lakehouse")
    variants.set(listOf("iceberg"))
    addons.set(listOf("hadoopAws"))
    managedConfigurations.set(listOf("implementation"))
    platformVersion.set("0.1.1-SNAPSHOT")
    platformImage.set("ghcr.io/openprojectx/spark-platform")
    localPlatformImage.set(false)
}
----

[cols="1,1,2", options="header"]
|===
| Property | Default | Description

| `line`
| `spark3`
| Selects the managed version-catalog bundle. Supported lines in this repository are `spark3`, `spark3-scala213`, `spark4`, and `cloudera`.

| `profile`
| empty
| Optional curated image profile name. When set, the default application base
image tag is `<line>-<profile>-<platformVersion>`. The application should still
set the variants and addons it compiles against.

| `platformVersion`
| `project.version`
| Version of `org.openprojectx.spark.platform:platform-bom`.

| `platformImage`
| `ghcr.io/openprojectx/spark-platform`
| Base image repository used by Jib for application images.

| `imageTag`
| `<line>-<variant-or-profile>-<platformVersion>`
| Base image tag used by Jib for application images.
Multi-word variant and addon ids keep lower camel case in Gradle bundle names
and are lowercased in rendered image tag segments, for example `hadoopAws`
becomes `hadoopaws`.

| `officialBuild`
| CI-derived
| Chooses whether user-selected `sparkPlatform` dependencies are exposed through
`implementation` or `compileOnly`.

| `managedConfigurations`
| `implementation` locally, `compileOnly` for official builds
| Configurations that extend `sparkPlatform` and `sparkPlatformBom`. Set this
when using Spark Platform like a BOM for scopes such as `api`,
`testImplementation`, or `testCompileOnly`.

| `localPlatformImage`
| `true` only for direct `jibDockerBuild`, otherwise `false`
| When enabled, Jib reads the selected platform base image from the local Docker
daemon by using a `docker://` base image reference. Use this for integration
builds that first build the platform image locally.

| `variants`
| empty list
| Optional runtime variants. The plugin uses these to add strict constraints,
JVM option rules, and the default application base image tag when no profile is
set. The `platform-image` module also accepts them through the
`sparkPlatform.variants` Gradle property.

| `addons`
| empty list
| Optional supplemental platform dependency packs. Addons add strict
constraints and image jars. Explicit custom image tags include addons; curated
release tags normally use a profile name instead of listing every addon.
|===

== Gradle Properties

[cols="1,2", options="header"]
|===
| Property | Description

| `sparkPlatform.line`
| Default Spark line for plugin and platform image builds.

| `sparkPlatform.managedConfigurations`
| Comma-separated Gradle configurations that should receive Spark Platform
constraints, for example `api,testImplementation`.

| `sparkPlatform.officialBuild`
| Forces official mode when set to `true`.

| `sparkPlatform.localPlatformImage`
| Forces application Jib builds to use the local Docker daemon platform image
when set to `true`.

| `sparkPlatform.imageRepository`
| Target repository for `:platform-image`.

| `sparkPlatform.imageTag`
| Target tag for `:platform-image`.

| `sparkPlatform.baseImageRepository`
| Source repository for the Spark base image. Defaults to
`ghcr.io/openprojectx/spark` for every supported Spark line.

| `sparkPlatform.baseImageSuffix`
| Optional suffix for the Apache Spark base image. Defaults to
`-java17-python3-r-ubuntu`.

| `sparkPlatform.variants`
| Comma-separated variants, for example `iceberg,hudi`. When omitted, platform
image defaults are read from `gradle/spark-platform-image.toml`.

| `sparkPlatform.addons`
| Comma-separated addons, for example `hadoopAws,hadoopGcs,icebergAws`. Addons are
platform-owned dependency packs that are not Spark/Scala compatibility
dimensions.

| `sparkPlatform.profile`
| Curated platform image profile, for example `lakehouse`. Profiles expand to
variants and addons from `gradle/spark-platform-image.toml` and use a short
profile tag.

| `sparkPlatform.imageConfig`
| Optional path to the platform image config file. Defaults to
`gradle/spark-platform-image.toml`.
|===

* OpenLineage `1.39.0`

* Hadoop AWS `3.4.2`

Spark SQL Kafka `spark-sql-kafka-0-10` is part of the line-managed Spark

`org.apache.kafka:kafka-clients` to `3.9.1`, so Spark Kafka's transitive Kafka

artifact uses the selected Scala binary suffix and its version follows the
selected Spark line.

The `cloudera` line is intentionally narrower than the Apache Spark lines. It
models the Cloudera Spark TCK surface: Spark core/sql/hive, Scala 2.12.15,
SLF4J API 1.7.36, Iceberg Spark 3.3, Hadoop AWS, Hadoop GCS, and Iceberg AWS
bundle. It does not try to mirror every Apache Spark variant/profile.

== Version Catalog Bundle Names

The plugin and platform modules derive bundle names from the Spark line and
variant/addon name instead of branching on specific Spark versions.

[source,text]
----
spark-platform-<line>-managed
spark-platform-<line>-variant-<variant>
spark-platform-<line>-addon-<addon>
spark-platform-<line>-variant-<variant>-managed
----

Examples:

[source,text]
----
spark-platform-spark4-managed
spark-platform-spark4-variant-iceberg
spark-platform-spark4-addon-hadoopAws
spark-platform-spark5-variant-legacy-managed
----

To add a new Spark line, add the corresponding aliases and bundles to
`gradle/libs.versions.toml`. For example, a future Spark 5 line would use
`spark-platform-spark5-managed`.

Use a `spark-platform-<line>-variant-<variant>-managed` bundle when a variant
cannot share the line-wide BOM. For constraints, the plugin and BOM can include
isolated variant-managed bundles alongside other requested variants because the
user still opts into concrete platform dependencies explicitly. Platform images
still split incompatible Scala variant groups instead of forcing them into one
image.

== Platform BOM

The `platform-bom` module publishes a Java Platform BOM.

For direct use without the plugin:

[source,kotlin]
----
dependencies {
    implementation(platform("org.openprojectx.spark.platform:platform-bom:0.1.0-SNAPSHOT"))
    implementation("org.apache.spark:spark-sql_2.13")
}
----

Direct BOM use is an advanced escape hatch. The preferred contract is the
plugin, where the application selects `line`, `variants`, and optional
`addons`, then adds versionless `sparkPlatform(...)` dependencies for the
platform APIs it uses.

When using the plugin, selected bundles contribute strict constraints to
`sparkPlatform`. The `sparkPlatformBom` configuration remains available when a
build intentionally wants to add a published BOM, but normal application builds
should not need it.

The plugin and BOM publish strict dependency constraints from the version
catalog. This keeps transitive dependencies from moving a Spark line away from
the selected platform version.

To force a transitive dependency version, add that transitive module as a
version-catalog alias and include it in the owning bundle. For example,
`sparkKafkaClients` is included in the line-managed and base-runtime bundles so
both application resolution and Spark base image assembly use
`org.apache.kafka:kafka-clients:3.9.1`.

Bundle scope summary:

[cols="1,1,1,1", options="header"]
|===
| Bundle | Gradle compile/runtime effect | Image packaging effect | Use for

| `spark-platform-<line>-managed`
| Adds strict constraints for the selected Spark line. It does not declare app
dependencies by itself.
| No direct packaging. Matching jars belong in the Spark base image through
`spark-base-<line>-runtime`.
| Spark, Scala, Hadoop, and line-managed Spark modules such as Spark SQL Kafka
and Spark Avro.

| `spark-platform-<line>-variant-<variant>`
| Adds strict constraints when the variant is selected.
| `platform-image` resolves and layers selected variant jars into
`/opt/spark/jars`.
| Optional Spark-compatible runtime families such as Iceberg, Hudi, Paimon, and
OpenLineage.

| `spark-platform-<line>-addon-<addon>`
| Adds strict constraints when the addon is selected.
| `platform-image` resolves and layers selected addon jars into
`/opt/spark/jars`.
| Optional reusable support packs such as Hadoop AWS, Hadoop GCS, or Iceberg
AWS.

| `spark-base-<line>-runtime`
| Not consumed by application builds.
| `spark-base-image` resolves and layers the clean Spark line runtime jars into
the Spark base image.
| The base Spark runtime image contents.
|===

== Image Layers

Spark Platform uses three runtime image layers for normal application delivery:

[cols="1,1,1", options="header"]
|===
| Layer | Built by | Contains

| Spark base image
| `spark-base-image`
| `/opt/spark` layout plus Gradle-managed Spark line runtime jars: Spark, Scala,
Hadoop, Spark SQL Kafka, Spark Avro, Kubernetes/YARN/MLlib/REPL modules, and
related baseline transitive jars.

| Platform image
| `platform-image`
| Starts from the Spark base image and adds selected variant/addon jars under
`/opt/spark/jars`, for example Iceberg, Hudi, Paimon, OpenLineage, Hadoop AWS,
Hadoop GCS, or Iceberg AWS.

| Application image
| user project with the Spark Platform plugin and Jib
| Starts from the selected platform image and adds only application classes,
resources, and application-owned libraries. Spark Platform-owned jars are
provided by the lower layers.
|===

The internal layout image used by `spark-base-image` is a build input only. It
contains the Spark distribution layout with Spark jars stripped in the same
Docker layer, then the runtime base image adds catalog-managed jars with Jib.

== Platform Image

The `platform-image` module builds a platform image using Jib. It starts from
the configured Spark base image repository, Spark version, Scala binary version,
and base image suffix:

[source,text]
----
<base-image-repository>:<spark-version>-scala<scala-binary-version><base-image-suffix>
----

The default base-image repository is the project-owned clean Spark image
repository, so the default resolved image looks like:

[source,text]
----
ghcr.io/openprojectx/spark:<spark-version>-scala<scala-binary-version>-java17-python3-r-ubuntu
----

The suffix can be overridden with `-PsparkPlatform.baseImageSuffix=...`, but
the default is intentionally fixed to Java 17, Python 3, R, and Ubuntu.
Spark 3 uses Scala 2.12 artifacts by default. Spark 4 uses Scala 2.13
artifacts. Use `spark3-scala213` when you need Spark 3.5.8 with Scala 2.13; it
is a separate Spark line, not a suffix override on the Scala 2.12 `spark3`
line.

Only variant and addon jars are layered by `platform-image`. Spark, Scala,
Hadoop, SLF4J, snappy, and the standard `org.lz4` provider are treated as
base-image-provided dependencies and are not copied from platform-image Gradle
resolution output. The base image owns the runtime baseline, including the
`hadoopSpark3` and `hadoopSpark4` versions from the catalog.

Build the default Spark 4 platform images:

[source,bash]
----
env GRADLE_USER_HOME=/data/.gradle ./gradlew :platform-image:jibDockerBuildPlatformImages \
  -PsparkPlatform.line=spark4
----

The aggregate task can build individual variant images, combined profile
images, or both:

* `sparkPlatform.buildIndividualImages=true|false`
* `sparkPlatform.buildCombinedImages=true|false`

Direct aggregate builds default to both for local inspection. The release-all
task uses `buildIndividualImages=false` and `buildCombinedImages=true`, so the
published release surface is curated combined profiles instead of every
individual variant tag.

Default curated profiles, default variants/addons, base image defaults,
isolated variants, transitive excludes, and capability-resolution rules are read from
`gradle/spark-platform-image.toml`. Use
`-PsparkPlatform.imageConfig=<path>` to point platform image aggregate tasks at
another config file.

When adding another library family, add the dependency coordinates and
`spark-platform-<line>-variant-<variant>` or
`spark-platform-<line>-addon-<addon>` bundles to
`gradle/libs.versions.toml`, then add the variant/addon to the desired curated
profiles in `gradle/spark-platform-image.toml`. The build script should
not need to change for another dependency, base image, curated profile,
transitive exclude, or capability rule.

Build one explicit variant set:

[source,bash]
----
env GRADLE_USER_HOME=/data/.gradle ./gradlew :platform-image:jibDockerBuild \
  -PsparkPlatform.line=spark4 \
  -PsparkPlatform.variants=iceberg,hudi,paimon,openlineage
----

Build one explicit variant plus addon set:

[source,bash]
----
env GRADLE_USER_HOME=/data/.gradle ./gradlew :platform-image:jibDockerBuild \
  -PsparkPlatform.line=spark4 \
  -PsparkPlatform.variants=iceberg,openlineage \
  -PsparkPlatform.addons=hadoopAws
----

Build only one curated profile tag:

[source,bash]
----
env GRADLE_USER_HOME=/data/.gradle ./gradlew :platform-image:jibDockerBuildPlatformImages \
  -PsparkPlatform.line=spark4 \
  -PsparkPlatform.profile=lakehouse \
  -PsparkPlatform.buildIndividualImages=false \
  -PsparkPlatform.buildCombinedImages=true
----

Build only individual tags:

[source,bash]
----
env GRADLE_USER_HOME=/data/.gradle ./gradlew :platform-image:jibDockerBuildPlatformImages \
  -PsparkPlatform.line=spark4 \
  -PsparkPlatform.variants=iceberg,hudi \
  -PsparkPlatform.buildIndividualImages=true \
  -PsparkPlatform.buildCombinedImages=false
----

By default, the platform image uses:

* base image: `ghcr.io/openprojectx/spark:<spark-version>-scala<scala-binary-version>-java17-python3-r-ubuntu`
* target image: `ghcr.io/openprojectx/spark-platform`
* target tag: `<line>-<profile-or-variant-addon-selection>-<project.version>`
* extra jars path: `/opt/spark/jars`

Combined variant sets must use a single Scala binary version so they match one
Apache Spark base image. The aggregate task derives compatibility from the
Scala binary versions in the variant bundles, so incompatible variants are
built in separate compatible profile groups instead of being forced into one
image. For example, a Scala 2.12-only variant will not be combined with Scala
2.13 variants.

`jibDockerBuild` writes the image to the local Docker daemon. It does not create
a tar file in the build directory.

[source,bash]
----
docker images ghcr.io/openprojectx/spark-platform
docker inspect ghcr.io/openprojectx/spark-platform:spark4-iceberg-0.1.1-SNAPSHOT
docker run --rm --entrypoint sh ghcr.io/openprojectx/spark-platform:spark4-iceberg-0.1.1-SNAPSHOT \
  -c 'ls -1 /opt/spark/jars | sort'
----

The current Jib image tasks are not compatible with Gradle configuration-cache
reuse. The platform-image build marks `jib`, `jibDockerBuild`, and `jibBuildTar`
as incompatible so a normal invocation does not reuse a stale broken
configuration-cache entry. Gradle will still execute the image task each time
because Jib does not declare Gradle task outputs for `jibDockerBuild`, but this
does not mean the image is rebuilt from scratch: Gradle's build cache remains
enabled for other tasks, and Jib/Docker still reuse base image layers and Jib's
layer cache.

== Spark Base Images

The `spark-base-image` module builds project-owned clean Spark base images for
all supported Spark lines. These images preserve the Apache Spark distribution
filesystem layout for Spark Operator and Kubernetes use, but the final
`/opt/spark/jars` contents are resolved by Gradle from the platform catalog and
BOM.

Each supported line is built in three logical steps:

* a layout image downloads and verifies the Apache Spark distribution, extracts
  `/opt/spark`, and strips `/opt/spark/jars` in the same Docker layer
* feature layout images start from the stripped layout image and install OS
  features such as Python 3, R, or both
* runtime images start from the matching layout image and use Jib to add
  Gradle-managed Spark, Scala, Hadoop, and related runtime jars

Layout images are released independently through the manual `Base Images`
GitHub workflow because they download and verify the Apache Spark distribution
and perform heavy apt installs for Python/R variants. Runtime base images are
released by the normal release workflow before platform images are published, so
catalog-managed Spark, Hadoop, Scala, and runtime jar changes move with the
platform release.

Build all base images locally:

[source,bash]
----
env GRADLE_USER_HOME=/data/.gradle ./gradlew :spark-base-image:dockerBuildSparkBaseImages \
  -PsparkBaseImage.localLayoutImages=true
----

The `sparkBaseImage.localLayoutImages=true` switch makes Jib read layout images
from the local Docker daemon with `docker://`. Release builds omit that switch
and use already-published layout images from `ghcr.io/openprojectx/spark`.

Build and publish only layout images:

Build and publish only runtime base images from already-published layout

== Application Images

The plugin applies and configures the Jib plugin for application projects.

The application image base is:

[source,text]
----
<sparkPlatform.platformImage>:<sparkPlatform.imageTag>
----

Application images follow Spark's Kubernetes image contract. The plugin keeps
the Spark entrypoint as `/opt/entrypoint.sh`, stages the user application jar at
`/opt/spark/app/app.jar`, and exposes the app classes, app libraries, and
platform jars through Spark classpath environment variables. Use
`local:///opt/spark/app/app.jar` as the application resource in Spark Operator
`SparkApplication.spec.jars`.

Application projects do not need to repeat Spark, Hadoop, Scala, variant, or
addon jars in their own image; the platform base image owns those runtime
artifacts.

For `jibDockerBuild`, the plugin prefixes the base image with `docker://` so
the local Docker daemon is used and Jib does not require the platform base image
to exist in the remote registry. For `jib`, official builds keep the normal
registry reference because publishing should be based on the registry platform
image.

If a higher-level task such as `integration` or `release` will eventually run
`jibDockerBuild`, Gradle's initial task list may not contain `jibDockerBuild`.
Set `sparkPlatform.localPlatformImage = true` in that integration build, or pass
`-PsparkPlatform.localPlatformImage=true`, so Jib uses the platform image built
earlier in the same Docker daemon.

The default application image target is:

[source,text]
----
<project.group>/<project.name>:<project.version>
----

A Spark Operator application uses that image as the Spark container image:

[source,yaml]
----
apiVersion: spark.apache.org/v1
kind: SparkApplication
metadata:
  name: spark4-iceberg
spec:
  mainClass: "org.openprojectx.spark.platform.examples.spark4.Spark4IcebergExample"
  jars: "local:///opt/spark/app/app.jar"
  sparkConf:
    spark.kubernetes.container.image: "org.openprojectx.spark.platform.examples/spark4-iceberg:0.1.1-snapshot"
    spark.kubernetes.authenticate.driver.serviceAccountName: "spark"
  runtimeVersions:
    sparkVersion: "4.0.1"
----

== Spark 4 Iceberg Example

The runnable example is in `examples/spark4-iceberg`.

[source,bash]
----
cd examples
env GRADLE_USER_HOME=/data/.gradle ../gradlew :spark4-iceberg:run --no-configuration-cache
----

The example:

* selects `spark4`
* adds versionless `sparkPlatform` dependencies for Spark SQL and Iceberg
* receives Spark SQL, Hadoop, and Iceberg versions from the selected platform constraints
* uses a local Iceberg Hadoop catalog
* uses Gradle Java toolchains for JDK 17 execution

== JDK Requirements

Use JDK 17 for development and runtime checks. The examples configure Java 17
through Gradle toolchains rather than a machine-specific `org.gradle.java.home`.

== Troubleshooting

Missing bundle error::
  If Gradle reports that `spark-platform-<line>-managed` is missing, add the
  matching bundle to `gradle/libs.versions.toml`.

Dependency version appears in a build script::
  If the dependency is platform-owned, move the version to
  `gradle/libs.versions.toml`, select the matching `sparkPlatform.line`,
  `sparkPlatform.variants`, or `sparkPlatform.addons`, and add only the versionless coordinate to
  `sparkPlatform`. Application-owned libraries can still use normal Gradle
  dependency declarations.

IDE uses a newer JDK::
  Configure the Gradle JVM to JDK 17. Example compilation and `JavaExec` tasks
  use Gradle Java toolchains.

Java module access error while running Spark or Iceberg::
  The plugin applies Spark's Java module options to managed `JavaExec` tasks and
  Jib application image entrypoints. If a new runtime family needs additional
  options, add a JVM option rule in the core module.
